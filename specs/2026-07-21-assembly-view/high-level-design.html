Assembly Module

Decision and public API overview for a generic, read-only object assembly layer under multistorageclient.assembly. This document explains what we are building, why it is parallel to StorageClient, and where application-owned planning ends and MSC-owned byte execution begins.

1. Executive decision

Build a separate multistorageclient.assembly module. Applications construct an AssemblyClient with a user-supplied AssemblyPlanProvider, local object-source bindings, and service executors. Resolving a logical name returns a fixed, expiring AssemblyView. The module neither subclasses StorageClient nor registers an assembled backend as a StorageProvider.
What users receive One logical byte object

Range reads, a read-only stream, metadata, and atomic local materialization.

What users provide Planning and bindings

A plan provider plus locally trusted storage and service implementations.

What remains unchanged MSC storage core

Existing clients, providers, profiles, paths, and mutation APIs retain their behavior.

This preserves MSC's original purpose—unified access to physical storage—while adding an application-layer view that composes bytes already available through that access. The assembly service is an application capability, not a new storage backend and not necessarily an HTTP service.

descriptor metadata → complete plan → validation + frozen bindings → executable AssemblyView

2. Goals, scope, and non-goals

A logical object is the ordered concatenation of three explicit segment kinds. Keeping the kinds visible makes the contract understandable and prevents application semantics from leaking into MSC.

SegmentMeaningExecuted by
ObjectSegmentAn exact byte range from an existing physical object.A locally configured StorageClient binding.
ServiceSegmentOpaque bytes generated or transformed by an application service.A user-defined, locally configured ServiceSegmentExecutor.
InlineSegmentA small, strictly bounded literal carried in the plan.The assembly module without external I/O.

Goals

  • Expose one read-only logical byte stream over ordered multi-source segments.
  • Resolve a complete plan with mandatory expiration and caller-controlled refresh.
  • Support exact ranges, streaming, metadata, and bounded local materialization.
  • Allow any planning backend through an in-process provider interface.
  • Keep credentials, endpoints, and executable bindings under local application control.
  • Preserve truthful consistency: expiration by default, stronger pinning only when enforceable.

Non-goals

  • Define a universal HTTP or gRPC API for the planning service.
  • Teach MSC about specific file formats, media containers, schemas, rows, or samples.
  • Add write, delete, copy, sync, or presign semantics to assembled views.
  • Make the feature automatically appear in every existing MSC, FUSE, CLI, or fsspec path.
  • Let a remote plan choose credentials, profiles, arbitrary endpoints, or Python objects.
Requirements versus defensive design. The requirements explicitly call for configured sources and services, direct storage reads for object segments, transport-neutral service segments, optional source versions/validators, bounded execution, and no breaking changes to existing providers. They do not require public binding-revision or resource-policy hooks, a built-in network executor, an MSC-defined external service API, or format-specific executors. If implementation work proposes those additional safeguards or conveniences, they remain separately reviewable defensive design—not requirements.

3. Why a separate Assembly module

The decisive question is substitutability: can assembly honestly satisfy the existing abstraction without special cases? It cannot. Assembly is a derived view over several authorities, while the existing types model unified access to physical storage authorities.

Why not StorageProvider

  • A provider is a leaf authority for one physical namespace; assembly orchestrates several authorities.
  • The provider interface includes mutations that have no coherent meaning for a derived view.
  • Only StorageClient currently owns providers. A provider that owns other providers would reverse MSC's dependency and lifecycle model.
  • Logical identity and expiration come from the plan provider, not from any physical backend.

Why not StorageClient

  • StorageClient unifies storage access; it should not acquire application-specific content orchestration.
  • A special case in read would spread into info, open, list, cache, download, sync, and error behavior.
  • Assembly needs fixed generations, expiration, service execution, and multi-source atomicity—concepts absent from the storage façade.
  • A parallel module can expose honest read-only capabilities without pretending to support the entire mutable client surface.
Provider pattern is not StorageProvider inheritance. AssemblyPlanProvider is a narrow application extension point that returns logical descriptors and plans. Existing StorageProvider implementations continue to perform physical I/O behind StorageClient.

Assembly module owns

  • Descriptor-to-plan resolution and executable-view construction.
  • Plan validation and logical range calculation.
  • Expiration, consistency, and view lifecycle.
  • Bounded execution, ordering, and atomic publication.
  • Stable public behavior independent of backend transport.

Application owns

  • Logical namespace and file-format meaning.
  • Plan-provider implementation and backend protocol.
  • Source bindings and service-executor implementations wired at construction.
  • Credential, endpoint, and dependency lifecycle choices.

Bindings own

  • Physical object or service-operation interpretation.
  • Authentication and connection management.
  • Transport-specific validation and bounded retries.
  • Resource safety and any stable identity enforcement they advertise.

5. Public API at a glance

The common workflow is intentionally small: inject dependencies once, resolve one generation, then operate on its view. Detailed signatures, configuration fields, limits, and typed-error construction live in the Implementation & Test Plan.

An object source wraps an already configured StorageClient. A service executor is a concrete user-owned implementation of ServiceSegmentExecutor. MSC defines the in-process behavior of that SPI, not the application's external protocol. 

# Imports are omitted. Configure physical storage through ordinary MSC APIs.
raw_storage = StorageClient(
    StorageClientConfig.from_file(profile="raw-data")
)

# Existing physical storage is exposed through a normal MSC client.
raw_source = ObjectSourceBinding(
    client=raw_storage,
)

# User code implements MSC's ServiceSegmentExecutor SPI.
rewrite_service = MyServiceExecutor(
    client=my_service_client,
)

# User code also implements the control-plane provider SPI.
catalog = MyAssemblyPlanProvider(client=my_catalog_client)

assembly = AssemblyClient(
    plan_provider=catalog,
    object_sources={
        "raw": raw_source,          # selected by ObjectSegment.source_id
    },
    service_executors={
        "rewrite": rewrite_service, # selected by ServiceSegment.executor_id
    },
)

# resolve() obtains descriptor metadata and a complete plan. It returns only
# after plan validation, binding freeze, and capability computation succeed.
view = assembly.resolve(
    AssemblyReference(namespace="exports", key="run-42.bin")
)

# metadata is the AssemblyDescriptor for this fixed generation.
descriptor = view.metadata

# Execute one logical range or publish the whole object locally.
footer = view.read(ByteRange(offset=footer_offset, size=8192))
view.materialize("/tmp/run-42.bin")

# Refresh creates a replacement view; the original never changes.
new_view = assembly.refresh(view)
MyServiceExecutor is application code, not an MSC built-in type. It translates one ServiceExecutionRequest into the application's HTTP, gRPC, SDK, or local-service call. It never receives the complete AssemblyPlan, never chooses segment order, and returns only the byte stream for one ServiceSegment under the declared length, range mode, deadline, and cancellation contract.

What AssemblyView encapsulates

AssemblyView is more than metadata: it is the immutable execution context for one resolved generation. It keeps the identity and plan stable while delegating heavy scheduling and I/O to client-owned machinery.

Frozen identity Descriptor metadata

view.metadata returns the control-plane descriptor: logical reference, generation, size, expiration, and optional application metadata.

Executable snapshot Validated plan and bindings

The complete ordered AssemblyPlan and a frozen alias-to-binding snapshot validated before the view is returned.

Operation façade Logical ranges to bytes

Maps reads and materialization windows to plan segments, then delegates admission, dispatch, staging, cancellation, and assembly.

An AssemblyDescriptor is metadata only and has no byte operations. The executable view never owns a StorageProvider, replans the object, or refreshes itself. open() creates a reader pinned to the same descriptor and plan; refresh() creates another view. Client-wide scheduling, reader state, staging, and atomic publication are detailed in Executor, AssemblyView, reader, and materialization.

Plans contain only the aliases raw and rewrite; credentials, service URLs, authentication, and lifecycle stay in the local bindings. Dependencies are borrowed by default. The exact façade is documented in the API reference subsection, and error construction in the implementation error contract. Custom executor details are in the executor SPI.

6. Worked example: assembling a Parquet object

Parquet makes the byte flow concrete, but it is only an example. The planning application understands page and footer semantics; the assembly module sees an ordered list of opaque byte segments, declared lengths, and local bindings.

The format boundary is intentional. If a different application assembles JSONL, a media file, or an unknown future format, the AssemblyClient algorithm is unchanged. Only the plan provider and service executors know why each byte range belongs in that position.

7. Extension model

Application-owned planning enters MSC through a provider-style in-process interface. Users may back this interface with HTTP, gRPC, a database, a local manifest, or an existing SDK without changing AssemblyClient. 

class AssemblyPlanProvider(ABC):
    def resolve_descriptor(
        self, request: ResolveAssemblyRequest
    ) -> AssemblyDescriptor: ...

    def get_plan(
        self,
        descriptor: AssemblyDescriptor,
        *,
        deadline_monotonic: float | None = None,
    ) -> AssemblyPlan: ...

    # Optional: advertise support and return a replacement generation.
    def refresh(self, request: RefreshAssemblyRequest) -> AssemblyDescriptor: ...

    def close(self) -> None: ...

The provider promises

  • resolve_descriptor() returns metadata only: one exact reference, generation, size, and future expiration.
  • get_plan() returns that descriptor's complete ordered object plan, never a range-specific fragment.
  • Canonical bounded models and stable public failure classifications.

AssemblyClient promises

  • Expose no AssemblyView before complete-plan validation and binding freeze succeed.
  • Perform no source or service byte I/O while resolving or validating the plan.
  • Never silently substitute a generation or downgrade consistency.
  • Resolve aliases only through the frozen local registries.
  • Supply a reusable conformance suite for custom implementations.

No universal control-plane network API

MSC does not define endpoints such as POST /assemblies:resolve. Network shape, authentication, and retry policy belong to the provider implementation. The same rule applies to service fulfillment: MSC defines the in-process SPI, not server endpoints, payloads, or authentication. MSC also standardizes the optional language-neutral assembly-list/v1 artifact.

Planning and byte fulfillment are separate roles

AssemblyPlanProvider says which bytes form the object. A user-owned ServiceSegmentExecutor produces bytes for exactly one service segment under its declared length, range, deadline, and cancellation contract. It receives neither the complete plan nor authority to order segments. One backend may support both concerns, but the application injects separate adapters so the paths can have different authorization, scaling, retry, and lifecycle policies.

Core adoption has no network-adapter dependency

Custom providers and service executors are sufficient for the core feature. A reusable network convenience adapter would be a separately approved follow-up with its own generic transport contract; it is not a prerequisite for the Assembly engine or its end-to-end tests.

8. Key behavioral guarantees

These are the guarantees an API reviewer needs to evaluate. Their exact algorithms, limits, exception hierarchy, and test cases are specified in the Implementation & Test Plan.

Fixed and expiring views

A view holds its descriptor, one complete plan, frozen bindings, computed capabilities, and lifecycle state. Expiration is checked before each new operation. Refresh creates another view and never mutates the old one.

Truthful consistency

EXPIRATION_ONLY is the default and does not promise immutable source bytes. SNAPSHOT_PINNED is available only when every non-inline segment has enforceable stable identity.

Complete-plan validation

MSC validates the segment union, aliases, expiration, per-segment lengths, and checked global sum before any byte I/O or view exposure. Invalid or unknown input fails closed.

Bounded execution

Concurrency, memory, staging, inline data, result sizes, and deadlines are client-wide bounded. Whole-result services stage once per operation and are sliced locally.

Atomic public operations

A one-shot read and each reader call publish no partial result on failure. Materialization publishes atomically; earlier successful streaming calls remain an already-observed prefix.

Local trust boundary

Plans carry aliases, never credentials or arbitrary endpoints. Public errors are typed and redacted so backend messages, resources, and secrets do not cross the boundary.

9. Compatibility and implementation direction

Unchanged

  • StorageClient constructor and public methods.
  • StorageProvider ABC, factories, and third-party providers.
  • Existing storage profile schema and msc:// behavior.
  • Current pathlib, fsspec, CLI, sync, and FUSE behavior.
  • Go components in the initial release.

Additive

  • multistorageclient.assembly package and explicit construction.
  • Provider and service-executor SPIs plus an adopter conformance suite.
  • Versioned assembly-list/v1 artifact and golden vectors.
  • Python client, view, reader, models, and stable error contract.
Explicit trade-off: a clean parallel module is not automatically transparent to every existing StorageClient consumer. Consumers accepting binary streams can use AssemblyReader; consumers requiring a local path can use materialize. Storage-shaped or filesystem adapters are follow-up decisions driven by concrete demand.
