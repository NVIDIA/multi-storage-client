1. What MSC provides—and what your application provides

This guide describes a proposed multistorageclient.assembly API. It is an adoption contract, not a description of a feature already available in a released package.
MSC provides the assembly types, orchestration, and validation. Your application provides the plan and every application-specific implementation. Assembly remains a separate layer above existing StorageClients; it is not a StorageClient subclass or a StorageProvider.

Plans contain aliases and declarative segment data only. Do not place clients, credentials, profiles, endpoints, or executable objects in a plan, and do not access StorageClient private providers.

2. Provide metadata first, then one complete plan

Implement AssemblyPlanProvider when plans come from an application-owned catalog. If your application already has validated assembly-list/v1 batches, use StaticAssemblyListProvider instead. 

class AssemblyPlanProvider(ABC):
    @abstractmethod
    def resolve_descriptor(
        self,
        request: ResolveAssemblyRequest,
    ) -> AssemblyDescriptor:
        """Return metadata for one fixed, unexpired generation."""

    @abstractmethod
    def get_plan(
        self,
        descriptor: AssemblyDescriptor,
        *,
        deadline_monotonic: float | None = None,
    ) -> AssemblyPlan:
        """Return the complete ordered plan for that descriptor."""

    @property
    def supports_refresh(self) -> bool:
        return False

    def refresh(self, request: RefreshAssemblyRequest) -> AssemblyDescriptor:
        raise AssemblyCapabilityError()

    def close(self) -> None:
        pass

Application provider

# MyAssemblyPlanProvider is application code.
provider = MyAssemblyPlanProvider(...)

Your implementation maps application metadata into MSC descriptors and complete plans.

Portable static provider

# StaticAssemblyListProvider is supplied by MSC.
provider = StaticAssemblyListProvider.from_batches(batches)

Use this when the application already produces the portable list artifact.

  • resolve_descriptor() returns control-plane metadata only: reference, generation, size, expiration, and optional descriptive fields.
  • An AssemblyDescriptor has no read, open, or materialize operation.
  • get_plan() receives that descriptor directly and returns every ordered segment for the complete logical object—never a range-specific fragment.
  • The plan must describe the same reference, generation, size, and expiration as the descriptor.
  • When refresh is supported, it returns metadata for a replacement generation; it never mutates the previous descriptor or view.

3. Build one complete, ordered AssemblyPlan

A plan is the ordered concatenation of three public segment kinds. MSC treats their contents as bytes and does not interpret application formats.

SegmentUse it forSelected by
ObjectSegmentA byte range from an existing stored object.source_id → local ObjectSourceBinding.
ServiceSegmentBytes produced by an application-defined operation.executor_id → application-supplied ServiceSegmentExecutor.
InlineSegmentA small literal carried in the plan.No alias or binding.
 
plan = AssemblyPlan.create(
    schema_version="assembly-plan/v1",
    descriptor=descriptor,
    segments=(
        InlineSegment(kind="inline", data=b"HDR", length=3),
        ObjectSegment(
            kind="object",
            source_id="raw",
            resource="records/part-0001.bin",
            byte_range=ByteRange(offset=0, size=4_194_304),
            length=4_194_304,
        ),
        ServiceSegment(
            kind="service",
            executor_id="rewrite",
            operation="build-metadata",
            parameters=FrozenJsonObject.from_mapping({"layout": "v3"}),
            length=16_384,
            range_mode=ServiceRangeMode.WHOLE_RESULT,
        ),
    ),
)
  • Tuple order is the logical byte order. Do not add a second ordering field.
  • The checked sum of segment lengths must equal descriptor.size.
  • Every source_id and executor_id must match a local binding supplied to AssemblyClient.
  • In v1, the client-only object binding rejects source selectors and ENFORCE_DURING_READ validators; IDENTITY_ONLY metadata does not pin bytes. MSC can instead verify a checksum over the complete object segment.
  • A service revision supports stronger consistency only when the selected executor advertises and enforces that revision kind; MSC can also verify a checksum over the complete service result.
  • MSC validates the complete plan before returning a view or performing byte I/O.
assembly-list/v1 is the portable representation of the same descriptor-and-plan model. Applications may exchange or persist it, then construct StaticAssemblyListProvider. The exact artifact schema and limits are documented in the Implementation & Test Plan.

4. Bind stored-object aliases to existing StorageClients

raw_storage = StorageClient(
    StorageClientConfig.from_file(profile="raw-data")
)

# ObjectSourceBinding is provided by MSC.
raw_source = ObjectSourceBinding(
    client=raw_storage,
)

object_sources = {
    "raw": raw_source,  # matches ObjectSegment.source_id
}
  • Configure credentials, account, endpoint, root, and provider through the normal StorageClient APIs.
  • The plan selects only the alias raw; it cannot select or reconfigure the underlying authority.
  • The binding must be able to interpret every resource assigned to its alias.
  • StorageClients remain application-owned and are always borrowed by AssemblyClient; manage them through their existing public APIs.
  • Request stronger consistency only when every non-inline segment has enforceable evidence; in v1, an object segment needs a verified complete checksum.

5. Implement ServiceSegmentExecutor for application-produced bytes

MSC provides only the ServiceSegmentExecutor SPI. It provides no concrete executor. Your application owns the complete implementation, constructor, dependencies, and fulfillment behavior.

The required implementation surface is intentionally small:

class ServiceSegmentExecutor(ABC):
    @property
    @abstractmethod
    def range_modes(self) -> frozenset[ServiceRangeMode]: ...

    @abstractmethod
    def execute(
        self,
        request: ServiceExecutionRequest,
        *,
        context: SourceReadContext,
    ) -> ServiceResultStream:
        """Return bytes for exactly one ServiceSegment request."""
Input or resultUser-facing meaning
ServiceExecutionRequest.segmentThe single ServiceSegment selected by MSC.
requested_rangeA subrange only when the executor advertises STABLE_SUBRANGES; otherwise absent.
SourceReadContextThe operation deadline, cancellation signal, and maximum output-chunk size.
ServiceResultStreamA closeable stream of byte chunks for this one request.

Executor obligations

  • Process one ServiceExecutionRequest; never require the complete AssemblyPlan.
  • Do not choose segment ordering or read another plan alias.
  • Return exactly the declared complete length or requested subrange, according to the advertised range mode.
  • Honor the supplied deadline and cancellation signal, emit bounded byte chunks, and make the result stream closeable.
  • Keep advertised capability values stable for the executor's injected lifetime, and preserve stable-subrange behavior throughout each operation.
Range modeWhat your executor promises
WHOLE_RESULTReturn the complete declared result for the segment. MSC validates it before exposing a requested slice.
STABLE_SUBRANGESReturn exactly the requested subrange, and ensure all subranges refer to one stable logical result during the operation.

RetrySafety.NEVER is the safe default. Declare RetrySafety.SAFE only when your implementation can repeat the operation without changing its meaning. Internal scheduling and retry mechanics are specified in the Implementation & Test Plan.

This guide deliberately does not show a concrete executor class or prescribe how an executor produces bytes. The application supplies an already constructed object that satisfies this SPI.

6. Compose AssemblyClient at the application boundary

Treat dependencies as inputs to your composition function. This keeps it explicit that the executor implementation comes from the application rather than MSC. 

def create_assembly_client(
    *,
    plan_provider: AssemblyPlanProvider,
    raw_storage: StorageClient,
    service_executors: Mapping[str, ServiceSegmentExecutor],
) -> AssemblyClient:
    return AssemblyClient(
        plan_provider=plan_provider,
        object_sources={
            "raw": ObjectSourceBinding(client=raw_storage),
        },
        service_executors=service_executors,
    )


# Every dependency instance comes from the application composition root.
assembly = create_assembly_client(
    plan_provider=plan_provider,
    raw_storage=raw_storage,
    service_executors={
        "rewrite": service_executor,
    },
)
  • raw must match every intended ObjectSegment.source_id.
  • rewrite must match every intended ServiceSegment.executor_id.
  • If no plan contains service segments, pass an empty service_executors mapping and implement no executor.
  • Injected dependencies are borrowed by default; the application remains responsible for their lifecycle.
  • Unknown aliases, incomplete plans, and unsupported capabilities fail before resolve() returns a view.

7. Resolve once, then operate on AssemblyView

resolve_descriptor(request) → descriptor → get_plan(descriptor) → validate + freeze + capabilities → AssemblyView
reference = AssemblyReference.from_mapping(
    namespace="exports",
    key="run-42.bin",
    parameters={"variant": "training"},
)

with assembly.resolve(reference) as view:
    descriptor = view.metadata
    assert descriptor.generation == "g-193"

    tail = view.read(
        ByteRange(offset=descriptor.size - 8192, size=8192)
    )

    with view.open() as reader:
        if reader.seekable():
            reader.seek(descriptor.size - 4)
            trailer = reader.read(4)

    view.materialize("/tmp/run-42.bin", overwrite=False)
OperationCaller-visible behavior
resolveReturns only after descriptor lookup, complete-plan validation, binding freeze, and capability computation all succeed.
metadataReturns the view's immutable AssemblyDescriptor; the descriptor itself is not executable.
readReturns one logical byte range from the fixed view generation.
openReturns a read-only stream over the same validated generation and plan; call seekable() before seeking.
materializePublishes the complete logical object to a local path without exposing a partial successful result.
  • A view contains its descriptor, complete validated plan, frozen bindings, computed consistency/capabilities, and lifecycle state.
  • The view never resolves a different generation implicitly.
  • Closing a view prevents new operations through that view; it does not mutate another view.
  • Manage the parent AssemblyClient and application-owned dependencies at the composition boundary.

8. Handle expiration, refresh, and consistency explicitly

Expiration

  • Every descriptor has a mandatory future expires_at.
  • A new operation on an expired view raises AssemblyExpiredError.
  • An operation admitted while valid may finish after expiration.
  • There is no implicit refresh or generation fall-forward.

Consistency

  • EXPIRATION_ONLY is the default and does not promise immutable source bytes.
  • SNAPSHOT_PINNED is opt-in. In v1, every object segment needs a verified complete checksum; each service segment needs an executor-enforced revision or a verified complete checksum.
  • Unsupported stronger consistency fails before byte I/O.
old_view = assembly.resolve(reference)

try:
    data = old_view.read()
except AssemblyExpiredError:
    new_view = assembly.refresh(old_view)
    assert new_view is not old_view
    data = new_view.read()

Refresh returns a newly resolved and validated view. It never mutates the old one. If refresh fails, the old view remains unchanged; whether it is usable still depends on its original expiration and lifecycle state.

9. Troubleshooting and adoption checklist

What you observeWhat to check
AssemblyExpiredErrorResolve or refresh explicitly; do not extend or mutate the old view.
Resolve returns no viewCheck descriptor echo, expiration, complete-plan length, aliases, and requested consistency.
Unknown object aliasEnsure ObjectSegment.source_id matches a configured object_sources key.
Unknown service aliasEnsure ServiceSegment.executor_id matches an application-supplied executor.
Service length or protocol failureCheck the executor's advertised range mode, returned length, chunk contract, deadline, and cancellation behavior.
Stronger consistency is rejectedConfirm every object segment has a complete checksum and every service segment has either an executor-enforced revision or a complete checksum.

Before adopting the feature

  • Choose a custom provider or StaticAssemblyListProvider.
  • Return metadata-only descriptors and complete ordered plans.
  • Configure every StorageClient outside the plan and bind it by a local alias.
  • Implement and inject ServiceSegmentExecutor only when service segments are present.
  • Keep credentials, dependency objects, and executable behavior out of portable plans.
  • Exercise full-object, partial-range, open-reader, materialization, expiration, and refresh flows.
  • Request only consistency guarantees that the plan evidence, Assembly core, and selected executors can enforce together.
  • Close AssemblyClient, views, and readers predictably, and manage application-owned providers, executors, StorageClients, and related resources through their own public APIs.
The reusable provider/executor conformance suites and the complete failure, performance, and security test matrix belong to the Implementation & Test Plan.
