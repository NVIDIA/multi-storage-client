1. Implementation stance

Implement an independent Python assembly layer and a language-neutral list codec. The Python layer owns public contracts, expiration, complete-object validation, scheduling, and dispatch. Object segments use an injected existing StorageClient; service segments use a separate ServiceSegmentExecutor SPI implemented by the user. The initial core release requires no built-in HTTP adapter. Existing StorageClient and StorageProvider behavior is not modified.
Extension boundary AssemblyPlanProvider

Injected Python SPI returns one complete object plan for a fixed descriptor generation; it need not use the portable list artifact.

Execution boundary StorageClient + Service Executor

Object bytes use public MSC APIs; synthesized bytes use an independent service SPI.

Transport boundary User-owned adapter

HTTP, gRPC, an SDK, or a local call remains behind the user implementation of the service SPI.

Constraints inherited from the high-level design

  • No AssembledStorageProvider, Provider-to-Provider ownership, or access to a client's private provider.
  • No StorageClient subclass and no implicit registration in storage profiles, provider factories, or URI routing.
  • No MSC-defined HTTP/gRPC contract for the plan service; only the in-process provider SPI is standardized.
  • No plan-supplied credentials, absolute origins, storage profiles, or arbitrary executable callbacks.
  • Every descriptor and portable-list snapshot has a mandatory expiration. The default EXPIRATION_ONLY policy does not pretend optional selectors, validators, or operation revisions are enforced; SNAPSHOT_PINNED is an explicit stricter opt-in.
  • A read, materialization, or open-reader lifetime may cross expiration only if admitted while the descriptor was valid. New operations on that view fail until the caller explicitly resolves a new view.
  • Object, service, and inline segments remain an explicit public discriminated union; service transport adapters remain application-owned behind the public executor SPI.
  • The initial public API is synchronous, read-only, explicit, and importable from multistorageclient.assembly.

Requirement alignment and proposal boundaries

Explicitly required

Configured sources and services; direct StorageClient reads for object segments; transport-neutral service segments; optional source versions/validators; bounded execution; and no breaking changes to existing providers.

Defensive design, not a requirement

Requirements do not explicitly require binding_revision or resource_validator. V1 keeps ObjectSourceBinding client-only, scopes instruction caches to one client and frozen registry snapshot, disables logical-byte caching, and adds no public provider-specific resource-policy hook.

Separate optional follow-up

Requirements do not explicitly require a built-in HTTP executor, an MSC-defined external service API, or a format-specific executor. Any generic HTTP convenience adapter requires separate approval.

This is an implementation proposal, not implementation authorization. Public types, capability semantics, strict expiration behavior, consistency policies, inline limits, service capabilities, and product scale inputs are approval gates before core source code changes begin. The optional generic HTTP adapter contract is not a core approval gate.

3. Runtime flow from public entry point to bytes

The complete call path for assembly.read(reference, ByteRange(28672, 20480)) is:

Normalize the request AssemblyClient.read canonicalizes the reference and range, checks client state, and rejects limits before any provider, StorageClient, or service-executor call.
Resolve one valid descriptor generation The client calls AssemblyPlanProvider.resolve_descriptor once. Its AssemblyDescriptor is metadata only—reference, generation, size, expiration, and optional descriptive fields—and cannot read, open, or materialize bytes. The client rejects a substituted reference or generation and a missing, malformed, or expired expires_at. No view is observable yet.
Acquire the complete object plan The client calls get_plan for that exact descriptor. A provider may page its backend internally, but returns all ordered segments for the object and cannot substitute a different generation or a range-specific fragment.
Validate and publish the view The validator checks the segment discriminator, checked global length, hard inline limits, aliases, optional selectors/validators/operation revisions/checksums, and the selected consistency policy. It then freezes the resolved local source/service objects and computes capabilities. Only now does AssemblyClient.resolve return an executable AssemblyView containing the descriptor, validated complete plan, frozen bindings, consistency/capabilities, and lifecycle state. Refresh repeats this pipeline and returns another view.
Intersect the caller range Only after global validation, the planner intersects the requested range with object and inline segments and chooses subrange or whole-result execution for each service segment.
Dispatch bounded operations Object segments call existing StorageClient APIs directly. Range-capable services receive stable subranges. Each touched WHOLE_RESULT stage key—descriptor generation, plan digest, and segment ordinal—gets one logical request submission per top-level operation and is staged under fixed memory/temp-file bounds before slicing; a SAFE executor may make multiple private transport attempts.
Verify every chunk Each binding enforces only capabilities present in its concrete contract: the initial object binding rejects selectors/enforcing validators, while service executors validate advertised revision and retry semantics. A checksum may be absent, but when supplied MSC always stages and verifies the complete segment before exposing any slice.
Commit atomically Verified chunks scatter incrementally into a private destination buffer; only after every chunk passes may the executor return immutable bytes. V1 publishes no logical-byte cache entry.

Failure boundary

A terminal error cancels queued work, signals the shared cancellation token, closes open service result streams, discards every unpublished chunk and staged output, and returns one stable domain exception. The token is internal in v1: MSC signals it after a sibling failure or parent-client close; the synchronous public façade has no caller-supplied cancellation input. The absolute operation deadline strictly bounds admission and MSC-owned waits. A provider or service executor receives that deadline and must honor it cooperatively. The current synchronous StorageClient.read accepts neither a deadline nor a cancellation token, so MSC can check the deadline before and after the call but cannot preempt an in-progress read. Applications must configure the StorageClient's underlying transport timeouts and retry envelope accordingly; a late result is discarded after the deadline is observed. A future additive object-binding contract may offer stronger cancellation without changing StorageClient.

Expiration is an admission check, not an in-flight cancellation deadline. An operation admitted while the descriptor is valid may finish after expires_at. Its result may be published to the caller or atomically materialized, but no instruction-derived cache entry is usable after expiration. A completed materialized file has an independent, application-owned retention lifetime and does not become invalid merely because its source descriptor later expires.

Public models live in Python, use frozen slotted dataclasses, and are validated at construction and again at trust boundaries. assembly-list/v1 is a normative, versioned, language-neutral JSON representation; providers may construct the equivalent models directly, but they do not get a private Python-only semantic contract.

Descriptor is not view. AssemblyDescriptor is immutable control-plane metadata and has no byte-I/O methods. AssemblyView.metadata returns the exact descriptor captured by that view.

Language-neutral list and canonical plan digest

An assembly list is one or more ordered AssemblyListBatch values with one list_id, issuance, mandatory expires_at, batch count, contiguous zero-based batch indexes, and exactly the declared number of batches. An object definition cannot be split between batches. This gives streaming traversal a bounded batch envelope while allowing an AssemblyListStore to choose a memory, disk, or remote exact-name index. That executable list is distinct from the optional browsing index: browse pages contain names/metadata/cursors and cannot be passed to the executor. The v1 artifact intentionally carries no refresh locator: refresh auth/transport is provider-owned through AssemblyPlanProvider.refresh. StaticAssemblyListProvider is non-refreshable; a custom provider may keep an artifact-carried or out-of-band locator internally and must still return a new identity/expiration.

plan_digest is a provider-supplied canonical instruction digest that MSC recomputes and verifies; the AssemblyPlan.create convenience computes it for direct-model providers. It is not a required expected digest of output bytes. It is sha256:<64 lowercase hex characters> over the UTF-8 RFC 8785 JSON Canonicalization Scheme projection containing exactly schema_version, descriptor, and the ordered discriminated segments. It excludes itself and execution-only deadlines. The list codec uses UTC RFC 3339 timestamps with exactly six fractional-second digits and Z, rejects timezone-naive values and unknown fields/kinds, and represents inline bytes as canonical unpadded base64url. FrozenJsonArray and FrozenJsonObject preserve JSON kind identity even for pair-shaped values; service parameters require an object root, while reference parameters allow scalar values only. Both parameter forms reject non-finite numbers and normalize negative zero. All object-key tuples use RFC 8785 UTF-16 code-unit order in memory and on the wire; lone surrogates are rejected and no Unicode normalization is performed. Frozen scalar equality/hash includes JSON kind plus canonical value: booleans differ from numbers, while 1/1.0 and 0/-0.0 normalize equal. Every non-JSON schema integer requires type(value) is int, rejecting bool and integral float. Every integer is in [-(2**53)+1, (2**53)-1]; offsets, lengths, sizes, and batch indexes are additionally non-negative.

Canonical invariants not expressed by the type declarations

  • Every AssemblyReference is unique within one complete list snapshot, including across batch boundaries.
  • A zero-size object has an empty plan. Otherwise, every segment length is positive and the checked sum of ordered segment lengths equals the descriptor size.
  • ObjectSegment.byte_range.size == ObjectSegment.length; InlineSegment.length == len(InlineSegment.data).
  • AssemblyDescriptor.etag is an optional identity of the complete assembled byte sequence. It is distinct from a source validator and MSC never synthesizes it from segment identities.
  • batch_digest is SHA-256 over the RFC 8785 canonical batch projection excluding batch_digest itself. plan_digest similarly excludes itself and execution-only deadlines.
  • Operation deadlines do not participate in model equality, hashing, digest projections, or cache identity.
  • plan_digest and batch_digest protect instructions/artifacts. Neither is an output-byte checksum nor sufficient evidence for SNAPSHOT_PINNED.

Local binding authority and hard inline limits

Object source and service aliases are local. A list cannot choose a profile, client, endpoint, credentials, or executor implementation. Validation resolves each alias and freezes the concrete local object in the view. An object resource is interpreted only through the bound StorageClient's existing public logical-path semantics; it cannot reconfigure that client or select authority outside it. Applications must therefore bind a least-privilege logical namespace appropriate for every plan they accept. V1 adds no provider-specific narrower-path policy on top of that public StorageClient namespace. Rebinding an alias never mutates an existing view. Inline data is decoded under two immutable v1 upper ceilings: at most 16 KiB per InlineSegment and 64 KiB total decoded inline bytes per object/plan. Client configuration may lower, but never raise, either ceiling. Encoded-size and overall-plan limits apply in addition; inline bytes never become a bulk-data escape hatch.

Contract choices

  • resolve_descriptor and get_plan are separate provider calls so metadata pins one descriptor generation before plan retrieval. resolve_descriptor returns only an AssemblyDescriptor; it never returns an executable object or exposes read, open, or materialize.   • AssemblyCapabilities.refresh captures the provider's immutable exact-bool supports_refresh value at construction. refresh rejects false before a provider call; StaticAssemblyListProvider advertises false, and a refreshable provider overrides both property and method.
  • No provider binding revision is required or echoed by the descriptor. Core instruction caches are scoped to one AssemblyClient and its concrete provider instance; they are never reused across provider instances.
  • get_plan returns one complete object plan. Backend/list batching is an implementation detail; a partial segment page never enters the validator or executor.
  • StaticAssemblyListProvider is the reference concrete-list surface. For each object definition it sets descriptor.generation = snapshot.list_id and descriptor.expires_at = snapshot.expires_at; reference, size, ETag, content type, and publication time come from that object definition. Exact canonical bytes exclude newline framing, but measurement streams into a capped counter/digest sink using the smaller remaining per-batch/aggregate allowance; it aborts at one over and never materializes a full encoding only to measure. Large deployments inject a replayable AssemblyListStore.
  • An AssemblyListStore declares one immutable snapshot, exact lookup, and traversal. Its conformance suite verifies stable results, fully validated batch provenance, close behavior, and the approved cost/memory envelope.
  • Browsing is a separate optional BrowsableAssemblyPlanProvider capability. Its pages and cursors are stable metadata views, not fragments of executable assembly lists.
  • resolve_descriptor, get_plan, and optional list are unconditionally thread-safe in v1; close begins only after the application stops new operations.
  • The synchronous v1 contract matches the current StorageClient surface. A later async API must use separate classes/methods, not silently change blocking behavior.
  • Provider retries are internal and bounded. AssemblyClient does not layer general retries around provider calls.
  • There is no automatic refresh. AssemblyClient.refresh(view) asks the provider for a replacement and returns a distinct view; a None consistency argument preserves the old view's originally requested minimum, while only an explicit override changes it. Failure leaves the old view unchanged: a still-valid, open view remains usable, while an expired or closed view remains unusable.

6. Validation and range planning

Configuration validation

Every public config/limits dataclass validates in __post_init__ and raises AssemblyConfigurationError before dependency/network I/O. Integer fields reject bool/integral float; all byte/count/concurrency/glob/timeout ceilings are positive except plan_cache_entries >= 0, and every integer configuration value is at most 2**53 - 1. Every seconds field is finite, within [0, 604800] where zero is allowed or (0, 604800] otherwise. Inline limits may only lower 16/64 KiB and object aggregate is at least segment. The exact byte-limit relationships are materialization_window_bytes <= max_read_bytes; max_object_read_chunk_bytes <= min(max_read_bytes, materialization_window_bytes, max_inflight_object_bytes, max_complete_segment_memory_bytes, max_staging_memory_bytes_per_operation, max_staging_memory_bytes_total); max_service_chunk_bytes <= min(max_read_bytes, materialization_window_bytes, max_complete_segment_memory_bytes, max_staging_memory_bytes_per_operation, max_staging_memory_bytes_total); max_complete_segment_memory_bytes <= max_complete_segment_staged_bytes <= max_staging_bytes_per_operation; and max_complete_segment_memory_bytes <= max_staging_memory_bytes_per_operation <= max_staging_bytes_per_operation. The complete-segment thresholds apply equally to WHOLE_RESULT service segments and to checksum-bearing object or service segments that require complete-result staging. Twice each of max_read_bytes and materialization_window_bytes fits max_output_memory_bytes_total. Per-operation staging memory and bytes do not exceed their matching client-wide totals. A staging path must be an existing absolute directory after os.fspath; construction neither creates nor write-probes it. Any future generic HTTP adapter defines and validates its own transport limits in a separately approved contract; those limits are not part of the Assembly core configuration. Every canonical string rejects lone surrogates before cache/digest work. Named namespace/key/resource/operation, cursor, and parameter limits apply first; max_identity_metadata_utf8_bytes is the catch-all for generation/list ID, aliases, ETag/content type, selector/validator/revision kind/value, listing generation, and any other canonical identity/metadata string without a stricter named cap. This semantic validation applies equally to decoded and live/direct provider models. List input requires 1 <= page_size <= max_list_page_size and a cursor no larger than max_cursor_utf8_bytes. Provider output must contain at most the requested item count, use a next cursor under the same byte ceiling, and pass normal descriptor validation before caching or glob accumulation. Glob uses the longest literal prefix, requests no page larger than the remaining max_glob_scanned_items budget, and raises AssemblyLimitExceededError without another provider call if a non-null cursor remains after the exact budget. It separately raises before yielding result max_glob_results + 1; earlier iterator yields are not rolled back.

Validation order

  1. Validate reference, schema/list versions, mandatory expires_at, and configured limits. Require descriptor.reference == request.reference and, when request.generation is non-None, descriptor.generation == request.generation; reject a silent fall-forward before plan/source I/O. Text limits count strict UTF-8 bytes; semantic parameter depth includes the parameter root object, while codec depth includes the complete document root (default 64 versus semantic 32). Recursive item count includes object members/array elements, and parameter bytes use exact RFC 8785 encoding.
  2. At operation admission, compare one captured UTC clock value with expires_at; equality is expired. Reject before provider plan lookup when a cached descriptor is already expired.
  3. Validate that the complete plan echoes the descriptor, then perform cheap structural checks before canonical encoding: exact model types, segment tuple count, discriminator, strict integers, text byte limits, parameter depth/item/canonical-byte limits, inline decoded limits, and checksum/revision grammar. Reject immediately on the first bound violation.
  4. Tuple order is logical order. Validate every positive length/object range with checked arithmetic; compute the checked sum of every segment length and require it to equal descriptor.size, including for a partial caller read.
  5. Stream the exact RFC 8785 digest projection (schema version, descriptor, ordered segments; no digest/deadline) into both SHA-256 and a capped byte counter. Do not construct a second complete encoding. Equality with max_plan_bytes is accepted; abort as soon as byte max_plan_bytes + 1 would be emitted, then verify plan_digest. The same bounded path handles decoded and direct-model providers.
  6. Resolve object aliases to existing StorageClients and service aliases to ServiceSegmentExecutors, then freeze those concrete local objects. Instruction caches remain scoped to this client and frozen registry snapshot; v1 exposes no binding identity for cross-registry reuse.
  7. Validate service modes/retry declarations, binding support for optional selectors/validators/operation revisions/checksums, staging feasibility, computed capabilities, and the selected AssemblyConsistency requirements.
  8. Normalize and bounds-check the caller's half-open logical range, intersect it with the already validated full plan, then allocate or stage output and begin I/O.

Range arithmetic

logical interval = [offset, offset + size)  ·  segment-local interval = [intersection start, intersection end)

Python checks every canonical integer and addition against the interoperable 2**53 - 1 ceiling before allocation or dispatch. A zero-length logical read returns b"" without segment I/O, but plan validation still occurs when required to prove the descriptor's complete-object definition and global length. A user executor owns any conversion to its transport's range convention.

No cross-segment coalescing in v1

Every intersected object segment remains one independent logical object operation. Within it, MSC issues consecutive public StorageClient.read calls of at most max_object_read_chunk_bytes; a checksum-bearing segment uses its complete source range as fetch_range and records the caller intersection in selected_range_in_fetch, while an ordinary segment fetches only the caller intersection and selects that complete fetch. Inline slices copy directly; STABLE_SUBRANGES receives exact intersections; each WHOLE_RESULT segment is keyed by descriptor generation, validated plan digest, and segment ordinal, and is staged once per admitted top-level operation.   • AssemblyClient defensively copies separate object_sources: Mapping[str, ObjectSourceBinding] and service_executors: Mapping[str, ServiceSegmentExecutor]; malformed or duplicate aliases fail construction. Either mapping may be empty when the resolved plan uses none of that segment kind.
  • Plans select only aliases. They cannot construct a client/executor or mutate endpoint, profile, prefix, credentials, or retry policy.
  • The complete public object binding is ObjectSourceBinding(client=storage_client). MSC freezes that concrete association in each view. The plan's resource is a logical path inside that client's already configured public namespace; it cannot reconfigure or expand the client authority. Applications bind a least-privilege namespace suitable for every accepted plan. V1 adds no provider-specific narrower path policy.
  • StorageClients are always borrowed and never closed by AssemblyClient. Calls to each distinct StorageClient object are conservatively serialized across aliases and views.
  • snapshot_revision_kinds defaults empty. A kind may be advertised only when the executor enforces that revision and it is sufficient, with operation and parameters, to pin the complete result bytes.
  • range_modes, snapshot_revision_kinds, and thread_safe are immutable for the injected executor's lifetime.   • The same client or executor may appear under multiple aliases. A transferred service executor is closed at most once; a StorageClient remains borrowed under every alias.
  • The same Python object identity cannot occupy different dependency roles (plan provider, service executor, StorageClient). A backend serving multiple roles uses distinct adapters; construction rejects cross-role identity reuse so ownership and CloseFailure remain unambiguous.
  • The plan provider and service executors are borrowed by default. close_dependencies=True transfers only those dependencies to AssemblyClient; it never transfers a StorageClient.
  • On failure AssemblyClient signals the shared cancellation token, attempts every service-stream close and shared-stage cleanup, and cancels queued work. A user executor observes that token through SourceReadContext; custom blocking I/O remains cooperative and must obey the context deadline. Failed stage artifacts follow the retained-reservation/orphan rules below.
  • Every yielded item is non-empty bytes with length ≤ SourceReadContext.max_chunk_bytes (from AssemblyClientConfig.max_service_chunk_bytes). Wrong type, empty, or oversized chunks attempt stream close and raise the primary ServiceProtocolError before copy/staging.
  • Exact allowlisted ServiceSegmentError, AssemblyCancelledError, and AssemblyTimeoutError outcomes are copied into fresh canonical instances using only validated safe fields. Checks outside the catch give cancellation/deadline precedence, and BaseException is never wrapped.
  • Cleanup attempts executor-owned result streams first in reverse registration-token order, then MSC-owned stage sink/fd/path resources in reverse registration-token order. A primary execution error takes precedence. With otherwise successful bytes, the first result-stream close failure becomes redacted ServiceExecutorError and the current private output is discarded. Object selector and validator semantics

In default EXPIRATION_ONLY mode, an ObjectSegment without a selector or validator is legal and is read directly through one or more bounded StorageClient.read(resource, Range(offset, size)) calls. Service range, staging, stability, and retry

  • STABLE_SUBRANGES is allowed only when both the segment and executor explicitly support it. Every requested subrange must belong to one stable logical byte string for the duration of the operation.
  • WHOLE_RESULT is the default. Within one resolved view and top-level operation, MSC submits one logical ServiceExecutionRequest through one execute call for each stage key (generation, plan_digest, segment_ordinal), where the ordinal is the zero-based position in the validated AssemblyPlan.segments tuple, even when several requested slices touch it. This key is operation-local and does not authorize cross-provider or cross-binding cache reuse. Different ordinals remain distinct even when every segment value is equal. Executor-owned SAFE retry may turn that submission into multiple private transport attempts.
  • A checksum may be omitted, but when supplied it is normative in every consistency mode. For a partial STABLE_SUBRANGES access, request/stage the complete segment, verify the digest, and only then slice; never compare the complete checksum with a partial response. Another enforcing revision does not waive this instruction.
  • max_complete_segment_memory_bytes is a per-complete-segment memory threshold and max_complete_segment_staged_bytes is the per-complete-segment hard cap. Before launch, reserve declared length against both per-operation and client-wide stage-byte totals. Use memory only when both memory totals also have capacity; otherwise use a unique private file. Multi-budget acquisition is all-or-none; temporary contention waits/serializes until the operation deadline and maps to AssemblyTimeoutError, while a complete segment larger than an applicable per-segment, per-operation, or client-wide hard cap fails before I/O.
  • AssemblyClientConfig.staging_directory selects the optional private-file parent for any complete-segment staging, including WHOLE_RESULT service output and checksum verification. A user executor may impose a stricter result limit, but it cannot weaken the AssemblyClient staging ceilings.
  • The result is length-checked while staging, optionally digest-checked, then sliced. Before external launch, each stage receives a deterministic registration token in logical scheduling order. Exact-length mismatch raises ServiceLengthMismatchError; complete checksum mismatch raises ServiceIntegrityError. Either failure attempts every MSC-owned stage sink/fd close and private-path unlink in reverse token order before bytes from that stage/current call are published; executor-owned result streams follow their separate rule above. Earlier successful reader calls may already be observed. Staged data is never reused across top-level operations in v1.
  • Private-stage allocation/create/write/flush/close/unlink failures use redacted AssemblyStagingError(cause_type=...); no private path or OS message is public. A cleanup failure never masks or chains onto an already-selected execution error: the primary typed error wins, while the failed artifact stays in a private orphan registry. If the operation would otherwise succeed, cleanup completes before returning bytes or publishing a destination; failure discards private output and raises AssemblyStagingError with the first safe failure type in deterministic cleanup order after every action was attempted. Reader close follows the same selection.
  • A file-backed stage reservation is released only after all owned sinks/file descriptors are closed and every private pathname is unlinked; an in-memory reservation waits for all owned sinks/buffers to release. Allocation/create failure before an artifact exists releases immediately. A partially cleaned artifact remains charged against the matching client-wide memory/disk total and is retried on reader/client close, preventing repeated failures from creating unbounded unaccounted storage. When an operation/reader ends, its per-operation charge ends only after ownership of the continuously held client-wide charge atomically transfers to the orphan ledger. Reader close marks the reader closed and releases its active-operation slot even when cleanup raises. Parent close attempts every retained artifact and adds at most one aggregate CloseFailure(dependency_kind="staging_artifact", aliases=(), ...) for the first safe failure type if any remain.
  • Destination failures use the exact redacted AssemblyMaterializationError(phase=..., cause_type=..., committed=...) contract. Before the atomic commit point committed=False and the prior/concurrently-created destination is preserved; cleanup or directory-fsync failure after publication reports committed=True and never falsely claims rollback.
  • RetrySafety.NEVER permits exactly one private attempt once the logical submission starts, regardless of whether that attempt produces any response bytes. RetrySafety.SAFE permits only the executor's documented bounded retry under the shared deadline; the assembly executor adds no outer retry.
  • Service operation revisions are optional. Identity-only revisions may classify metrics but cannot satisfy SNAPSHOT_PINNED. An enforcing revision qualifies only when its kind appears in snapshot_revision_kinds; a fully verified complete-result checksum qualifies independently.

8. Executor, AssemblyView, seekable reader, and materialization

Client-wide admission and byte reservations

  • Acquire exactly one max_active_operations slot before provider/source I/O or large allocation for each top-level resolve, info, refresh, list page, one-shot read, materialization, or open. A glob iterator holds one from first advancement through exhaustion/close. A successfully opened reader holds its slot until reader/parent close; a passive view holds none.
  • Convenience methods propagate the admitted operation context into a private resolve/plan helper; that helper never reacquires. Glob's private page fetches reuse its iterator slot instead of calling the public admitted list path, and materialization windows/reader calls reuse their operation's existing slot. This rule is required for every convenience at max_active_operations=1.
  • Slot waiters allocate no large buffer, observe the shared deadline/cancellation, and are awakened by parent close. Capture the expiration admission clock only after slot acquisition; timeout maps to AssemblyTimeoutError, cancellation to AssemblyCancelledError, and parent close to AssemblyClosedError.
  • AssemblyGlobIterator is a public idempotently closeable context manager. It acquires on first advancement, and exhaustion, explicit close, iterator error, cancellation/deadline, or parent close releases exactly once; close before first advancement acquires nothing. Early-breaking callers use with or close(). Its own close/exhaustion makes later advancement return StopIteration; parent close maps later use to AssemblyClosedError.
  • Before allocating a read result or materialization window, reserve twice its clamped length from client-wide max_output_memory_bytes_total. Reject a single reservation above the total; otherwise wait to the deadline. Release after the immutable bytes cross to caller ownership or the window is written/discarded.
  • Immediately before launching each complete-segment stage, atomically reserve that stage's declared length against both per-operation and client-wide total staging bytes. Do not pre-reserve every stage in the plan. Select memory only if both memory budgets also have space; otherwise select a private file. Acquire all counters under one scheduler or none, never hold one budget while waiting for another. This permits forward readers and materialization to serialize individually feasible stages whose sum exceeds the per-operation cap; retained seekable-reader stages keep their reservations until close.
  • The independent client-wide max_inflight_object_bytes bounds live StorageClient result chunks. Thus client-owned output, stage memory/disk, object chunks, operation count, and active source calls all have non-multiplying global ceilings across callers.

Executor algorithm

  1. Receive one fully validated complete object plan plus the caller's requested range; expiration has already passed the operation-admission check.
  2. Create one SourceReadContext with the operation's absolute monotonic deadline and thread-safe internal cancellation token. MSC signals that token after a sibling failure or parent close; v1 has no caller-supplied cancellation input.
  3. Allocate one private pre-sized output buffer after all size/coverage limits pass.
  4. Intersect the range with the three segment kinds; group object operations by StorageClient binding, service operations by executor, and copy inline slices locally.
  5. Acquire a client-wide concurrency budget shared across concurrent views and reads.
  6. For each object operation, split fetch_range into consecutive chunks no larger than max_object_read_chunk_bytes. Before each public StorageClient.read, check the absolute deadline and reserve the declared chunk length from the client-wide max_inflight_object_bytes budget. The call has no deadline parameter and cannot be preempted; after it returns, check the deadline again, discard late bytes, require exact length, and release the reservation. Hold one concurrency permit and any per-client serialization lock across the ordered sequence. An ordinary operation fetches only its intersection. A checksum-bearing operation stages the complete fetch_range, hashes it, then copies selected_range_in_fetch, so temp-file staging never requires a complete StorageClient result in memory. Call range-capable services with exact subranges and key each WHOLE_RESULT stage by descriptor generation, validated plan digest, and segment ordinal.
  7. As each chunk or staged slice completes, verify exact length. When a checksum is supplied, first stage and verify the complete segment (never a partial body against a complete digest), then copy the requested slice into the non-overlapping private destination and release it when lifecycle rules permit.
  8. On first terminal error, stop scheduling, signal cancellation, attempt every service-stream/stage close and private-file unlink in reverse deterministic registration-token order, and drain cooperative provider/service work only to its required deadline. An already-running synchronous StorageClient call remains non-preemptible until it returns or its configured underlying timeout fires; keep its permit and byte reservation until then and discard any late result. Retain failed artifacts and their reservations for later retry, discard the private output, and preserve the original typed terminal error as the public outcome.
  9. After every destination slice succeeds, freeze to bytes and return; v1 commits no logical-byte cache entry.

The AssemblyClient semaphore is client-wide and bounds active StorageClient and service executions; a streaming service result retains its permit and any per-executor lock until staging completes or closes. Creating a per-read pool would allow concurrent callers × max_concurrency active executions. A user-owned executor may impose a stricter transport-specific limit, but that implementation detail neither replaces nor multiplies the AssemblyClient budget.

AssemblyView and reader state

  • An AssemblyView is the executable handle. It retains one metadata-only AssemblyDescriptor, the complete validated AssemblyPlan, frozen concrete source/service bindings, mandatory expiration, originally selected required_consistency minimum, computed consistency/capabilities, and lifecycle state. AssemblyView.metadata returns that exact descriptor. None of this executable state is exposed by AssemblyPlanProvider.resolve_descriptor.
  • It never mutates or refreshes itself. AssemblyClient.refresh(view) returns a new view and requires a replacement generation and expiration. With required_consistency=None, refresh preserves the old view's originally required minimum; it never derives the minimum from the computed consistency or a later client default. Only an explicit argument changes it.
  • Each public read, materialize, and open admission captures the clock once and rejects an expired view before range planning or source/service work. Once admitted, that operation may finish after expiration.
  • Concurrent view.read calls are supported; an individual seekable reader is not thread-safe.
  • An AssemblyReader is one admitted operation from successful open until close. It owns the pinned descriptor/validated plan, SourceReadContext, operation-local complete-segment stage registry, cursor, and closed state.
  • Closing the originating AssemblyView rejects new work through that view but does not cancel an already opened reader. The reader survives independently until its own close or parent-client close.
  • Reader read/seek calls do not recheck descriptor expiration or resolve another generation. A reader opened while valid may continue after expiration; a new open on that view fails.
  • Each reader read intersects its minimal logical range with the same validated complete-object plan. Any complete-segment stage required by a WHOLE_RESULT service or checksum-bearing object/service segment is created at most once per descriptor generation, validated plan digest, and segment ordinal for that reader and retained across seeks until close.
  • Capability computation is total. Validation rejects a WHOLE_RESULT or checksum-bearing full stage when its declared complete length exceeds its applicable per-result hard cap or max_staging_bytes_per_operation. Executor-advertised range modes are included in the computation but cannot weaken client limits. Every successfully resolved v1 view therefore has byte_range_read=True and streaming_open=True; a future false flag raises AssemblyCapabilityError before execution I/O.
  • Seek is true only when the checked sum of all distinct potentially retained stages is at most max_staging_bytes_per_operation. Equality is accepted. Seekable readers retain encountered stages; forward-only readers and materialization release a stage immediately after the cursor/last window passes its final logical byte, allowing several individually feasible stages to serialize even when their sum exceeds the cap. V1 never evicts and replays a stage behind a seekable cursor.
  • Using a view or reader after parent close raises AssemblyClosedError.
  • After non-negative/checked validation, explicit ByteRange reads clamp at EOF: zero size or offset at/past EOF returns empty, and a crossing range ends at object size. Negative/overflowing input raises InvalidAssemblyRangeError. Reader seeks beyond EOF are allowed; a target below zero raises that error, and subsequent reads beyond EOF are empty. The clamped output length must not exceed max_read_bytes.
  • Atomicity is per reader call, not the reader lifetime. read/readinto stages the current call privately; terminal execution failure publishes nothing for that call, leaves cursor and destination buffer unchanged, closes/poisons the reader, and raises the typed error. Prior successful calls remain an observable prefix; later read/seek raises AssemblyClosedError. Pre-execution argument errors do not poison it.
  • Override RawIOBase.read(size=-1) and readall(); do not inherit an implementation that loops through committing readinto calls. Compute the full clamped public-call length first (-1 means the remainder), reject it above max_read_bytes, stage it privately, and advance the cursor exactly once after complete success.

Materialization

view.materialize(
    destination: str | os.PathLike[str],
    *,
    overwrite: bool = False,
    fsync: bool = False,
) -> None

For a filesystem path, write and close a unique temporary file in the destination directory and validate the entire assembled output. Create every private materialization or complete-segment stage file exclusively, use no-follow creation where the platform supports it, and use POSIX mode 0600. Never reuse or follow an existing path. Publication retains the temporary inode's effective mode. With fsync=True, flush and fsync that file before publication. The commit point is one atomic filesystem operation: os.replace when overwrite=True; when false, create the destination with an atomic same-filesystem no-replace primitive (initially hard-link the closed staging inode, then unlink its temporary name). A precheck is only an optimization and never supplies race safety. If no atomic no-replace primitive is supported, fail with AssemblyMaterializationError(phase="commit", cause_type="OSError", committed=False) without changing the destination. A destination created concurrently wins and is preserved. An arbitrary binary stream cannot provide atomic replace; v1 should therefore keep path materialization as the guaranteed API and direct stream users to open(). If materialization began before descriptor expiration it may complete afterward. Before the atomic commit point, any error has committed=False, removes the staging name best-effort, and preserves the previous destination. After commit, the new destination is already visible; unlinking the extra staging link and optional parent-directory fsync occur afterward. Their failures use phase="cleanup" or "directory_fsync" with committed=True; MSC must not claim rollback, and durability is indeterminate after directory-fsync failure. Once commit succeeds, the file's retention and deletion are application policy: later descriptor expiration neither deletes nor invalidates completed bytes. All windows of one materialization share one operation context and complete-segment stage registry, so a WHOLE_RESULT service or checksum-bearing object/service segment ordinal is not re-executed or refetched. Every operation-local private stage is cleaned successfully before the destination commit; a cleanup failure therefore prevents publication rather than leaving a committed destination with an unreported private orphan. Each logical window is exactly min(materialization_window_bytes, remaining_bytes); the last may be shorter, and the private assembled-output buffer never exceeds the configured window (separate complete-segment staging reservations still apply).

Memory policy

  • read is bounded by max_read_bytes. The initial Python implementation may hold its mutable private output and immutable returned bytes simultaneously, so the specified peak is at most twice the clamped result length plus bounded in-flight chunks and service stages; a later proven no-copy path may lower but not raise this gate.
  • materialize uses windowed reads and a staging file so object size is not duplicated in memory.
  • Complete plans are bounded by segment count and encoded size before range intersection or read scheduling.
  • Complete-segment stages have separate per-result memory, per-result bytes, operation-wide, and client-wide temp-file byte ceilings. These limits cover WHOLE_RESULT services and complete object/service verification required by checksums; each descriptor-generation, plan-digest, and segment-ordinal stage key is materialized once per top-level operation.
  • Inline data has hard decoded upper ceilings of 16 KiB per segment and 64 KiB per complete object; configuration may only lower them.
  • Every user-owned service stream is measured against declared length and client staging ceilings; transport metadata is never trusted as a substitute.

Retry ownership

  • Provider adapter retries its own transport without changing the exact request.
  • StorageClient retains its documented retry behavior; AssemblyClient adds no outer object-read retry. Because the current read API accepts no deadline, applications configure its underlying transport timeout and retry envelope with the Assembly operation budget in mind.
  • A ServiceSegmentExecutor may create more than one private attempt only when the segment says RetrySafety.SAFE, using its documented bound under the shared deadline. RetrySafety.NEVER permits exactly one private attempt once submission starts, even when that attempt fails before any response byte is observed.
  • The assembly executor never retries an object or service submission. It owns concurrency, internal cancellation, staging cleanup, atomic publication, and hard deadline enforcement for admission and MSC-owned waits; it can only observe the deadline before and after a synchronous StorageClient call.
  • No layer retries invalid/expired plans, binding mismatches, enforced revision mismatches, digest failures, unknown kinds, or protocol violations.
  • Defaults prevent multiplicative retries: every nested operation receives one absolute monotonic deadline and each WHOLE_RESULT stage key has one logical executor submission. Only its explicit SAFE policy may create multiple private attempts inside that submission; NEVER always remains one attempt.
  • AssemblyTimeoutError (also TimeoutError) is the sole public outcome once the shared absolute deadline is observed exhausted at provider, scheduling, object, service, or staging boundaries. The deadline hard-bounds admission, MSC-owned waits, and cooperative provider/service work. An in-progress synchronous StorageClient call cannot be preempted and may return after that instant; MSC checks immediately afterward, discards late bytes/errors, and then raises AssemblyTimeoutError. While budget remains, a provider-local transient timeout maps to AssemblyProviderRetryableError, and object/service local timeouts map to their transport classes. The v1 façade has no caller-cancellation argument: its internal token is signaled by sibling failure or parent close, and cooperative code checks it between attempts or chunks.

Each structured leaf accepts only its documented bounded keyword fields, calls the no-argument base initializer, and never puts arbitrary caller text in args, str, or repr. At provider, validator, object-client, executor, stream-close, and dependency-close boundaries, only exact allowlisted public classes are recognized. Classification occurs inside except Exception, but the caught object is discarded and the fresh error is raised only after the handler exits. Its only explicit cause may be a newly constructed _SanitizedDependencyCause, whose sole field is safe_type_token. The original object must not be reachable through __cause__, __context__, args, attributes, or formatted traceback. BaseException control-flow signals are not wrapped or logged by this layer.

Lifecycle and ownership

  • StorageClients are always borrowed because the existing public StorageClient API has no close lifecycle. AssemblyClient never attempts to close one.
  • close_dependencies=False also borrows the plan provider and service executors.
  • With close_dependencies=True, close the provider and each distinct service executor only. A Static provider honors its separate close_store flag.
  • Close sequence: reject new work, cancel queued operations, signal and close active service result streams/internal iterators, drain cooperative work to its deadline, and wait for any already-running synchronous StorageClient call to return under its configured underlying timeout. Then attempt every retained staging artifact, clear caches/views, and optionally close dependencies.
  • Close is state-idempotent: the first call permanently rejects new work. It retries retained artifacts in reverse deterministic registration-token order, then attempts all owned dependencies in deterministic plan-provider then service-executor first-alias order. Every artifact/dependency is attempted; at most one leading staging_artifact record captures the first safe cleanup failure type, then each distinct dependency appears once with sorted/deduplicated aliases. If cleanup fails, the client remains closed and raises one redacted AssemblyCloseError. A later close() retries only outstanding orphan artifacts and never re-closes an already attempted dependency; once no artifact remains, subsequent calls are no-ops.
  • Live clients/views/readers are non-pickleable by default; only explicit configuration/factory wrappers may reconstruct them.
  • Expiration does not close an in-flight operation or a successfully materialized file. It rejects subsequent operations and invalidates instruction-derived caches.

Telemetry

  • Spans: assembly.resolve, assembly.plan, assembly.read, assembly.object.read, assembly.service.execute, and assembly.materialize.
  • Metrics: list batches/objects, plan segments by kind, inline bytes, object/service bytes, core service-executor submissions, complete-segment staging bytes/backend, amplification, expiration rejection, cooperative cancellation latency, cache hit/miss, and failure class.
  • Low-cardinality labels only: segment kind, configured alias, range mode, result class, and staging backend.
  • Executor-internal retries and transport-specific telemetry remain owned by the user implementation because the core SPI does not expose them.
  • No logical keys, resources, credentials, ETags, or auth revisions as metric labels. Trace attributes are redacted and bounded.

Security controls

  • Treat provider output as untrusted and validate before allocation or I/O.
  • Limit list batch/object count, logical object size, one read, segment count, encoded bytes, decoded inline bytes, resource/parameter length, source/service result range, complete-segment staging memory/temp bytes, concurrency, retries, and time.
  • Use separate local object/service aliases and frozen local binding objects; instruction caches never cross an AssemblyClient or frozen registry snapshot, and logical-byte caching is disabled in v1.
  • Interpret object resources only through each least-privilege bound StorageClient's public logical namespace. Service executors confine operations to their locally configured authority; transport-specific controls belong to that implementation.
  • Create every MSC-owned private stage/materialization file exclusively, use no-follow creation where supported, and use POSIX mode 0600; publication retains that inode's effective mode.
  • Make all errors/reprs safe by construction and test redaction with canary secrets.

Initial performance gates

  • Blocking product-input gate: before implementation acceptance, product owners approve the maximum object count, maximum segments or encoded list bytes, exact lookup/traversal mix, and process memory envelope. Gate 0 cannot close with any value marked TBD.
  • test_large_list_full_traversal_stays_below_approved_latency_and_rss_limits passes at the approved maximum using numeric Gate 0 thresholds.
  • test_large_list_exact_lookup_hit_and_miss_stay_below_approved_latency_limits passes with the selected AssemblyListStore implementation.
  • test_large_list_batch_decode_peak_rss_stays_below_approved_bytes proves batches do not accumulate accidentally against a concrete byte limit.
  • test_approved_max_segments_and_encoded_plan_bytes_meet_latency_and_memory_envelope passes at both exact approved boundaries and rejects one over before I/O.
  • Gate 0 records the permitted planner time ratio between the approved N and 2N segment cases; the benchmark fails when that numeric ratio is exceeded.
  • Observed concurrent source reads never exceed the configured client-wide ceiling.
  • Observed top-level operations, output-memory reservations, stage-memory/stage-disk reservations, and in-flight object bytes never exceed their client-wide limits under concurrent read/open/materialize/glob workloads; max_active_operations=1 conveniences complete without nested-admission deadlock.
  • Object request count equals the sum of ceil(selected_source_bytes / max_object_read_chunk_bytes) per logical object operation: caller-intersection bytes normally, complete segment bytes when a checksum is supplied. V1 performs no cross-segment coalescing.
  • Peak Python read memory is bounded by up to twice the clamped returned length plus configured in-flight object/service chunks and complete-segment staging limits, not all unexecuted bytes.
  • WHOLE_RESULT core submission count is exactly one execute call per touched operation-local (generation, plan_digest, segment_ordinal) per top-level operation; this key is not reused across providers or bindings. Executor-internal SAFE attempts belong to that executor's own conformance tests. Successful cleanup removes private files; an injected unlink failure remains charged, is retried, and surfaces through the specified redacted error/close record rather than being reported as success.
  • Failure cancellation closes/drains queued work, internal waits, and cooperative service streams within the remaining operation deadline. A synchronous StorageClient.read already in progress is measured separately and is not claimed to be preemptible by Assembly.

Local benchmark throughput absolute numbers are recorded but do not block early pull requests because shared CI is noisy. After Gate 0 records an approved same-machine regression percentage, nightly measurements beyond that numeric threshold trigger investigation rather than an automatic functional failure.
